    Extended Features

    In addition to its syntactic features, CKSP offers various optimizations and extended functionalities designed to enhance workflow and improve flexibility when structuring projects. These features include fast compile times, advanced import mechanisms, and global preprocessor conditions.

    Fast Compile Times

    Beyond the language's syntactic features, CKSP provides fast compilation. With the compiler implemented in C++, it achieves over 97% faster compile times on large codebases compared to the SublimeKSP plugin. This dramatically reduces development time, especially for complex KSP projects, leading to an improved workflow.

    Pragma Directives

    CKSP lets you control key compiler settings right in your source via two #pragma directives. These always apply project-wide (i.e. across imports), but they can be overridden by command-line flags.

    #pragma output_path

    Specify exactly where the compiled output should go:

    #pragma output_path("path/to/file.txt")

    output_path file extension

    Note that the file extension must be *.txt.

    A later #pragma output_path (even in an imported file) will override an earlier one.

    To facilitate the transition from SublimeKSP and use this #pragma directive with both compilers, I have added a temporary workaround: simply prefix the directive with a single-line comment (with no whitespace between the slashes and #pragma). For example, by writing //#pragma output_path(...) CKSP will recognize and process the directive, while SublimeKSP will ignore it. This approach is especially helpful for projects migrating between compilers or maintaining compatibility with both.

    1
2
    {#pragma save_compiled_source Samples/resources/scripts/my_script.txt}
//#pragma output_path("Samples/Resources/scripts/my_script.txt")

    In this example, the first line uses SublimeKSP syntax and is ignored by CKSP, while the second line is recognized by CKSP but ignored by SublimeKSP.

    #pragma optimize

    0.0.7

    Override the global optimization level for the project. This is useful for debugging, performance tuning or reducing compile times. The optimization level can be set by using one of the following arguments to the #pragma optimize directive:

    #pragma optimize("none")
    No optimizations are applied.

    #pragma optimize("simple")
    Simple optimizations are applied, including constant folding and dead-code elimination.

    #pragma optimize("standard")
    This is the default optimization level, which includes optimizations like constant folding, constant propagation, constant expression propagation, variable pruning and dead-code elimination.

    #pragma optimize("aggressive")
    This is the most aggressive optimization level, which includes all optimizations from the standard level and applying more iterations of the optimization passes. This increases compile times.

    Command-Line Arguments

    When using CKSP from the command line, you can specify the described #pragma directives as command-line arguments. This allows you to set the output path and optimization level without modifying the source code. However, the optimization command-line arguments will be overridden by any #pragma directives in the source code.

    Flag Effect
    --optimize=none, -O0 No optimization
    --optimize=simple, -O1 Simple optimizations
    --optimize=standard, -O2 Standard optimizations (default)
    --optimize=aggressive, -O3 Aggressive optimizations
    -o <file>, --output <file> Set output file

    A -o/--output flag always takes precedence over any #pragma output_path. If neither CLI nor pragma is used, CKSP defaults to standard optimization and an output file of <input_dir>/out.txt.

    Enhanced Import Path Flexibility

    The CKSP compiler supports importing code from separate *.cksp files using the import <filepath> syntax. Import paths can be specified relative to the file containing the import statement. This allows for greater flexibility in project structuring, as import statements in any *.cksp file can refer to other files based on their location. This feature simplifies module management and the organization of complex projects.

    Global Preprocessor Conditions

    The scope of preprocessor directives, such as SET_CONDITION and RESET_CONDITION, is extended to the project level rather than being limited to individual files. This means that a condition set in one file, such as DEV_STATE, can influence the inclusion or exclusion of code blocks with the USE_CODE_IF or USE_CODE_IF_NOT directives across all imported *.cksp files. This global scope makes it easier to manage project environments and conditional code throughout the entire project.
